# Onboarding UI rename, Home State column, and datepicker fix

Three changes bundled in one diff: the public-facing onboarding form is renamed from "Onboarding Application" to "Candidate Information Sheet," the internal candidate list replaces the Work Site column with a Home State column, and a bug is fixed where editing a candidate failed because the mat-datepicker received a raw date string instead of a `Date` object.

Watch for: The Home State column is displayed and sortable in the candidate list, but no form field exists to collect it from users (confirmed). Sorting on `homeState` will coerce `undefined` to the string `"undefined"`, producing incorrect sort order for candidates without the field set (confirmed). The fallback dummy data in `getDummyCandidates()` does not include `homeState`, so the column will render blank for all fallback rows (confirmed).

## High-level view

The naming change is purely cosmetic — labels, headings, button text, and placeholder text all shift from "Onboarding" language to "Candidate Information Sheet" language. The Work Site field is removed from the public form entirely (both the HTML input and the form control / validator), and `workSite` becomes optional in the submission payload.

The Home State column introduces a new optional field on the `Candidate` model and wires it into the list's template, search, and sort. However, the field has no data-entry path: neither the public form nor the internal add/edit modal exposes a `homeState` input. The mock data hard-codes `'TX'` for every row, masking the fact that real usage will always produce `undefined`. The column will be permanently empty until a form input or API-derived value is added.

The datepicker fix constructs a `Date` object from the stored ISO date string when editing a candidate. The `'T00:00:00'` suffix anchors parsing to midnight local time, avoiding the timezone-shift issue where `new Date('2025-03-15')` is parsed as UTC midnight and may display as the previous day in western-hemisphere timezones.

## Home State column without a data-entry path

The `Candidate` interface gains an optional `homeState?: string` field. The candidate-list template renders `{{ candidate.homeState }}`, the search function matches against `(c.homeState || '').toLowerCase()`, and clicking the column header sorts by `'homeState'`. The mock interceptor's `buildMockCandidates()` hard-codes `homeState: 'TX'` for every row.

But there is no form control named `homeState` anywhere — not in the public onboarding form (which now omits Work Site entirely) and not in `AddCandidateModalComponent`'s `basicInfoForm`. The candidate-list component does pass `homeState: result.basicInfo.homeState || undefined` in the edit-save handler, but `basicInfoForm` never defines that control, so `result.basicInfo.homeState` will always be `undefined`.

The practical effect: real candidates created through either form path will never have a `homeState` value. The column exists in the UI but will always be blank unless the backend independently populates the field or a future form change adds the input.

## Sort coercion on undefined homeState

The generic sort in `applyFiltersAndSort` indexes the candidate object by column name:

```typescript
const aVal = a[column];
const bVal = b[column];
```

For an optional field like `homeState`, `aVal` can be `undefined`. The sort function only checks for `boolean` and `string` types explicitly; `undefined` falls through to the else branch, which does `String(aVal).localeCompare(String(bVal))` — producing `"undefined".localeCompare(...)`. This puts candidates without a state after entries like `"TX"` but before entries starting with `"V"` or later, which is arbitrary and confusing. The search filter already uses the defensive `(c.homeState || '')` pattern; the sort should do the same.

## Datepicker startDate initialization

In edit mode, the original code passed `candidate.startDate` (a string like `'2025-03-15'`) directly as the form control value. Angular Material's datepicker expects a `Date` object for its value binding; receiving a string leaves the internal picker state invalid, which causes the form to remain in an invalid state and blocks submission.

The fix constructs `new Date(candidate.startDate + 'T00:00:00')`. Appending `'T00:00:00'` forces the date to be parsed as local time rather than UTC, which avoids the off-by-one-day display issue in timezones behind UTC.

## Naming consistency and Work Site removal

The public form removes the Work Site field completely — the HTML input, the `workSite` form control, the `Validators.required` on it, and the inclusion in the submission payload. All user-visible strings on the public form now say "Candidate Information Sheet" or "Information" instead of "Onboarding Application." The internal generate-link dialog follows suit. The button in the candidate list is shortened to "Generate Info Sheet Link."

One residual: error messages in `getTokenErrorMessage()` still say "onboarding link" (e.g., "This onboarding link has expired"). These are pre-existing strings not touched in this diff, so out of scope, but worth noting for a follow-up if full terminology consistency is desired.
